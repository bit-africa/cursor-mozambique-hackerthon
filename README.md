## About the Event

| | |
|---|---|
| **Event type** | One-day developer build challenge (hackathon format) |
| **Theme** | Flood preparedness, response & recovery in Mozambique |
| **Core tool** | Cursor — used live for planning, coding, debugging & refactoring |
| **Organised by** | BIT Academy |
| **Venue** | Faculdade de Engenharia, Maputo |
| **Date & Time** | Saturday, 4 July 2026 · 10:00 AM |

The 2025/26 rainy and cyclone season has been one of the worst in a generation. This hackathon asks participants to ship a working prototype that helps a real community — built mobile-first, in Portuguese, for low-bandwidth conditions.

Goals:
- **Showcase Cursor** as a force-multiplier — proving how much a small team can build in a single day when AI handles the heavy lifting.
- **Produce genuinely useful ideas** — prototypes that could be developed further into tools for responders, volunteers, and at-risk communities.

## The Challenge & Tracks
 
Teams pick **one of four tracks**. Each track keeps the theme tight while leaving room for creativity:
 
| Track | Focus | What teams might build |
|-------|-------|------------------------|
| **1&nbsp;·&nbsp;Alerta** | Early warning | Crowd-sourced or sensor-fed water-level reporting that pushes warnings to at-risk neighbourhoods. Bonus: works over SMS / USSD or fully offline. |
| **2&nbsp;·&nbsp;Rota&nbsp;Segura** | Evacuation | Map of flood zones, nearest shelters and safe routes. Bonus: live capacity tracking per evacuation centre. |
| **3&nbsp;·&nbsp;SOS&nbsp;/&nbsp;Reunião** | Response | Report trapped people, match relief needs (clean water, shelter, meds) to volunteers and donors, or help separated families find each other. |
| **4&nbsp;·&nbsp;Depois** | Recovery | Track cholera / disease hotspots, map clean-water points, or manage crop and livestock loss claims for fair aid distribution. |
> **Scope guidance:** ship one feature that works end-to-end rather than five half-built screens. Flooding is a "does it actually function under pressure" theme — the judging rewards exactly that.

## Requirements & Rules

### Hard Requirements (judged)
- **Built in Cursor** — teams must show their workflow (Composer, chat, agents, rules) and prove it with a Cursor dashboard screenshot.
- **Any stack** — any language, framework or platform is welcome. Cursor is the only required tool.
- **Mobile-first** and usable on a slow 3G connection.
- **Portuguese UI** (or local language; English acceptable only as a fallback).
- **A working demo** — live or recorded. Not slides.
- **Public repository** + a 3-minute pitch.

### Rules
1. The project is built during the event. Coming with an idea, notes or generic boilerplate is fine — a materially pre-built project is not.
2. The prototype must address a real flood-related problem relevant to Mozambique.
3. Any language, framework or platform is allowed. Open data, public APIs and mock data are all welcome.
4. Cursor must be used actively throughout the build.
5. Every team presents a live or recorded demo within the time limit.
6. Teams: 2–4 people.

## Resources

### Free APIs & Data (optional helpers)
Pair one real API with one mock fallback so a flaky connection or rate limit never blocks your demo.

| Service | Use | Why it fits |
|---------|-----|-------------|
| [Open-Meteo](https://open-meteo.com) | Weather / rainfall | Free, no API key. Historical + forecast rainfall — ideal for flood-alert logic. |
| [OpenStreetMap](https://www.openstreetmap.org) + [Leaflet](https://leafletjs.com) | Maps | Free, open maps. Pair with Nominatim for free geocoding. No billing setup. |
| [Africa's Talking](https://africastalking.com) | SMS / USSD / Voice | Built for African networks — perfect for offline-reach alerts. Free sandbox. |
| [HDX (Humanitarian Data Exchange)](https://data.humdata.org) | Open datasets | Population, admin boundaries, prior flood data for Mozambique. |
| [ReliefWeb API](https://reliefweb.int/about/api) | Live situation data | Real-time reports & figures on the current flood response. |
| [Twilio (trial)](https://www.twilio.com) | SMS / WhatsApp | Alternative messaging with free trial credit; good WhatsApp support. |

## Judging Criteria

Each criterion is scored 1–5, then multiplied by its weight for a total out of 100. The weighting deliberately favours real impact, working software and genuine use of Cursor.

| # | Criterion | What we reward | Weight | Max |
|---|-----------|---------------|--------|-----|
| 1 | **Impact & Relevance** | Solves a real flood problem; understands local context (connectivity, language, who is affected). | x5 | /25 |
| 2 | **Functionality** | It works. The core feature runs end-to-end without breaking in the demo. | x5 | /25 |
| 3 | **Use of Cursor** | Clearly leveraged Cursor (Composer, agents, rules, multi-file edits); can explain the speed-up. | x4 | /20 |
| 4 | **Innovation** | A novel angle or clever approach — not a generic CRUD app. | x3 | /15 |
| 5 | **Code Quality** | Readable, reasonably structured code; sensible README. | x3 | /15 |

**Bonus points (up to +10):** Offline-first / works on USSD (+3), live data integration (+3), Portuguese-language UI (+2), accessibility features (+2).
